Servicios Finales (actualizados)

Config Server

	Centraliza la configuración de todos los microservicios.

	Maneja perfiles (local, dev, prod) desde un repositorio Git.

	Cada servicio obtiene su configuración al arrancar.

Eureka Server

	Registro y descubrimiento de servicios.

	Cada micro se registra con nombre y puerto.

	Facilita balanceo de carga y resiliencia.

API Gateway

	Punto de entrada único.

	Valida JWT en cada request.

	Enruta hacia el microservicio correspondiente.

	Aplica rate limiting, logging y tracing.

	Authorization Service (photo-app-auth-service)

	Login y emisión de JWT.

	Valida credenciales contra la tabla users.

	Consulta roles asociados al usuario (no a cuentas).

	Genera tokens JWT con userId y roles.

Users Service (photo-app-users-service)

	Registro y gestión de usuarios.

	CRUD de usuarios (crear, consultar, actualizar, desactivar).

	Al registrar un usuario, se le crea automáticamente al menos una cuenta activa (tipo BASIC).

	Expone endpoints para perfil de usuario.

	Flyway inicializa todas las tablas en photo_app_db.

Accounts Service (photo-app-accounts-service)

	Gestión de cuentas de usuario.

	Un usuario puede tener varias cuentas.

	CRUD de cuentas (crear, listar, activar/desactivar, cambiar tipo).

	Ya no maneja roles: solo atributos propios de la cuenta (accountName, accountType, activeAccount).

Albums Service (photo-app-albums-service)

	Gestión de álbumes de fotos.

	CRUD de álbumes (crear, listar, actualizar, desactivar).

	Relación 1:N entre cuentas y álbumes.

	Cada álbum pertenece a una cuenta específica.

Photos Service (photo-app-photos-service)

	Gestión de fotos dentro de un álbum.

	CRUD de fotos (subir, listar, eliminar, desactivar).

	Almacenamiento binario en AWS S3 bucket (o MinIO en local).

	La base guarda solo metadatos: nombre, URL, estado, timestamps.


En resumen: primero terminamos todos los servicios sin seguridad, luego Config Server y Gateway, y finalmente seguridad + tests.

Pensando en resiliencia (Resilience4j)
Cuando un microservicio llama a otro, siempre existe el riesgo de que el otro esté caído o lento.

Resilience4j te da circuit breakers, retries y fallbacks.

En este caso, si al registrar un usuario falla la creación de la cuenta, tienes que decidir:

Reintentar (si el fallo es temporal).

Fallback: por ejemplo, marcar al usuario como creado pero sin cuenta, y enviar un mensaje a una cola (Kafka, RabbitMQ) para crear la cuenta después.

Fail fast: abortar el registro y devolver error.
