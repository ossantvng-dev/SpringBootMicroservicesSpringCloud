📌 Funcionalidades por servicio
1. Discovery Service (Eureka)
Registro y descubrimiento de microservicios.

Permite al Gateway y a los servicios encontrarse dinámicamente.

No expone lógica de negocio, solo infraestructura.

2. API Gateway (Spring Cloud Gateway)
Punto único de entrada al sistema.

Rutas explícitas y limpias (/users-service/**, /accounts-service/**, /albums-service/**).

Filtros globales:

Validación JWT.

Logging y métricas.

Rate limiting.

Load balancing entre múltiples instancias de cada microservicio.

Reescritura de paths (RewritePath) para que los microservicios reciban rutas limpias.

3. Users Service
Endpoints principales:

POST /users → crear usuario.

GET /users/{id} → obtener usuario.

DELETE /users/{id} → borrar usuario.

GET /users/status/check → health check.

Seguridad con roles y authorities (@PreAuthorize).

Persistencia en MySQL.

Exposición de métricas vía Actuator.

Comunicación con otros servicios (ej. Accounts o Albums) vía Feign.

4. Accounts Management Service
Gestión de roles y permisos.

Endpoints:

POST /accounts → crear cuenta/rol.

GET /accounts/{id} → obtener cuenta/rol.

DELETE /accounts/{id} → borrar cuenta/rol.

Integración con Spring Security para roles a nivel de método.

Persistencia en MySQL.

Podría proveer authorities como PROFILE_DELETE, ADMIN, etc.

5. Albums Service
Endpoints:

GET /users/{id}/albums → obtener álbumes de un usuario.

POST /users/{id}/albums → crear álbum para un usuario.

Persistencia en MySQL.

Comunicación con Users Service vía Feign para validar existencia de usuario.

Exposición de métricas vía Actuator.

6. Config Server
Centraliza configuración de todos los servicios.

Usa repos Git para almacenar application-dev.properties y application-prod.properties.

Integra con Spring Cloud Bus + RabbitMQ para refrescar configuración en caliente.

Soporta encriptación/decriptación de propiedades sensibles (ej. credenciales DB, secretos JWT).

7. Infraestructura y soporte
RabbitMQ: mensajería/eventos entre servicios.

Resilience4J: circuit breakers, retries, rate limiting.

Micrometer + Zipkin: métricas y tracing distribuido.

ELK Stack: centralización de logs.

Docker: empaquetar y desplegar servicios.

Entornos DEV/PROD: configuración diferenciada en Config Server.

🚀 Conclusión
La nueva arquitectura tendrá:

Gateway como entrada única, con seguridad JWT y balanceo.

Users Service para gestión de usuarios.

Accounts Service para roles y permisos.

Albums Service para álbumes asociados a usuarios.

Config Server + Eureka como infraestructura base.

Observabilidad, resiliencia y despliegue en contenedores.

Servicios Finales:

1. Config Server
   - Centraliza la configuración de todos los microservicios.
   - Permite manejar perfiles (local, dev, prod) desde un repositorio Git.
   - Cada servicio obtiene su configuración al arrancar.

2. Eureka Server
   - Registro y descubrimiento de servicios.
   - Cada micro se registra con nombre y puerto.
   - Facilita balanceo de carga y resiliencia.

3. API Gateway
   - Punto de entrada único.
   - Valida JWT en cada request.
   - Enruta hacia el microservicio correspondiente.
   - Aplica rate limiting, logging y tracing.

4. Authorization Service (photo-app-auth-service)
   - Login y emisión de JWT.
   - Valida credenciales contra la tabla users.
   - Consulta roles asociados a las cuentas.
   - Genera tokens JWT con userId y roles.

5. Users Service (photo-app-users-service)
   - Registro y gestión de usuarios.
   - CRUD de usuarios (crear, consultar, actualizar, desactivar).
   - Al registrar un usuario, se le crea automáticamente al menos una cuenta activa.
   - Expone endpoints para perfil de usuario.
   - Flyway inicializa todas las tablas en photo_app_db.

6. Accounts Service (photo-app-accounts-service)
   - Gestión de cuentas de usuario.
   - Un usuario puede tener varias cuentas.
   - CRUD de cuentas (crear, listar, activar/desactivar).
   - Asociación de roles a cuentas.

7. Albums Service (photo-app-albums-service)
   - Gestión de álbumes de fotos.
   - CRUD de álbumes (crear, listar, actualizar, desactivar).
   - Relación 1:N entre cuentas y álbumes.

8. Photos Service (photo-app-photos-service)
   - Gestión de fotos dentro de un álbum.
   - CRUD de fotos (subir, listar, eliminar, desactivar).
   - Almacenamiento binario en AWS S3 bucket (o MinIO en local).
   - La base guarda solo metadatos: nombre, URL, estado, timestamps.
